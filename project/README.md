# Итоговый проект по курсу «Инженерия Искусственного Интеллекта»

## 1. Паспорт проекта

- **Название проекта:** `Прогнозирование оттока клиентов телеком-оператора`
- **Автор:** `Ким Андрей Павлович`
- **Группа:** `БББО-12-24`
- **Контакт:** `@Vanq5`

- **Краткое описание (2-4 предложения):**  
  `Проект посвящён построению сервиса прогнозирования оттока клиентов телеком-оператора. По профилю клиента (срок обслуживания, тип контракта, траты, набор услуг, обращения в поддержку) сервис оценивает вероятность ухода и категорию риска. Используются синтетические данные, классические ML-модели и ансамбли, проводится сравнение и обоснованный выбор финальной модели. Результат — REST API, который по профилю клиента возвращает вероятность оттока, категорию риска и ключевые факторы.`

---

## 2. Структура проекта

Проект организован в следующей структуре:

- `requirements.txt` – зависимости проекта (библиотеки Python, необходимые для запуска).
- `Dockerfile` – контейнеризация.
- `docker-compose.yml` – запуск одной командой.
- `Makefile` – удобные команды (make train / serve / test).
- `pytest.ini` – конфигурация тестов.
- `example_request.json` – готовый пример запроса к /predict.
- `report.md` – отчёт по проекту (постановка задачи, данные, эксперименты, результаты).
- `self-checklist.md` – чеклист самопроверки проекта перед сдачей.
- `notebooks/` – ноутбуки: EDA и эксперименты.
- `src/` – основной код проекта:
  - `data/` – генерация и загрузка данных;
  - `features/` – препроцессинг;
  - `models/` – обучение, эксперименты, трекинг;
  - `service/` – REST API сервис (Flask, запускаемый пакет);
  - `utils/` – конфигурация, логирование;
  - `train.py/` – точка входа: обучение.
- `data/` – демонстрационная выборка.
- `configs/` – конфигурация (YAML) и .env.example.
- `tests/` – тесты (pytest).
- `artifacts/` – сохранённые модели, отчёты, результаты экспериментов.

Структура соответствует базовой; точки входа оформлены как модули (`python -m src.train`, `python -m src.service`).

## 3. Требования и установка

### 3.1. Требования

- Python `>= 3.13` (рекомендуется, минимально поддерживается `>= 3.10`).
- `git` — для клонирования репозитория.
- Опционально: Docker — для запуска сервиса в контейнере.

### 3.2. Установка окружения

Примерный сценарий (можно адаптировать под ваш проект):

```bash
# Перейти в папку проекта
cd project

# Создать и активировать виртуальное окружение
python -m venv .venv
# Linux / macOS:
source .venv/bin/activate
# Windows:
.venv\Scripts\activate

# Установить зависимости
pip install --upgrade pip
pip install -r requirements.txt
```

## 4. Как запустить проект

### 4.1. Запуск обучения модели

```bash
cd project
source .venv/bin/activate          # при необходимости
python -m src.train
```

Команда сгенерирует датасет, обучит и сравнит модели, выберет финальную и сохранит артефакты в `artifacts/`: `model.joblib`, `metrics.json`, `model_comparison.csv`, графики в `plots/`, запись эксперимента в `experiments/`.

### 4.2. Запуск сервиса (REST API)

```bash
cd project
source .venv/bin/activate          # при необходимости
python -m src.service
```

Сервис поднимается на порту **8000** (настраивается через `SERVICE_PORT`).

**Эндпоинты:**

| Метод и путь | Назначение |
|---|---|
| `GET /` | описание API и пример запроса |
| `GET /health` | проверка работоспособности и готовности модели |
| `GET /metrics` | счётчики наблюдаемости (запросы, ошибки, латентность) |
| `POST /predict` | прогноз оттока по профилю клиента |

**Проверка работоспособности:**

```bash
# health-check
curl http://localhost:8000/health

# прогноз оттока
curl -X POST http://localhost:8000/predict \
  -H "Content-Type: application/json" \
  -d '{
    "senior_citizen": 0, "tenure": 2, "monthly_charges": 89.5,
    "total_charges": 178.6, "num_support_calls": 4,
    "partner": "No", "dependents": "No", "contract": "Month-to-month",
    "internet_service": "Fiber optic", "online_security": "No",
    "tech_support": "No", "streaming_tv": "Yes",
    "payment_method": "Electronic check", "paperless_billing": "Yes"
  }'
```

Пример ответа `/predict`:

```json
{
  "churn_probability": 0.9955,
  "churn_prediction": 1,
  "risk_category": "high",
  "model_name": "gradient_boosting",
  "key_factors": [
    {"feature": "contract", "label": "Тип контракта", "value": "Month-to-month", "importance": 0.2571},
    {"feature": "internet_service", "label": "Тип интернет-услуги", "value": "Fiber optic", "importance": 0.2347}
  ]
}
```

### 4.3. Запуск через Docker

```bash
cd project
docker build -t churn-prediction .
docker run -p 8000:8000 churn-prediction
```

Или одной командой через docker-compose:

```bash
docker compose up --build
```

Docker-образ обучает модель на этапе сборки, запускает сервис через production WSGI-сервер `waitress` под непривилегированным пользователем.

## 5. Данные

- **Источник:** синтетический датасет, генерируется модулем `src/data/generate.py`. Структура повторяет открытый датасет *Telco Customer Churn*; целевая переменная формируется по логит-модели с нелинейными эффектами.
- **Где лежат файлы:** `data/sample_customers.csv` — демонстрационная выборка (200 клиентов). Полный датасет (7000 строк) не коммитится — он детерминированно воспроизводится при обучении.
- Подробное описание — в `data/README.md`.

## 8. Ограничения и дальнейшая работа

**Текущие ограничения:**

- данные синтетические — модель не валидировалась на реальных клиентах;
- порог классификации фиксирован (0.5); под бизнес-задачу его стоит подбирать;
- метрики наблюдаемости хранятся в памяти и сбрасываются при перезапуске;
- интерпретируемость прогноза — на уровне глобальной значимости признаков (без пер-объектных объяснений вроде SHAP).

**Направления развития:**

- подбор порога классификации под стоимость ошибок (precision/recall trade-off);
- пер-объектные объяснения прогнозов (SHAP);
- экспорт метрик в формате Prometheus и дашборд мониторинга;
- хранение истории запросов и мониторинг дрейфа данных;
- расширение реестра моделей (CatBoost/XGBoost) и автоматический тюнинг.
